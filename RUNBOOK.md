# 42ctl — Operations Runbook

How to build, cut a release, rotate publish credentials, and revoke a bad release. `42ctl`
decrypts plaintext locally, so its supply chain is part of the vault threat model — treat every
publish path as a security control, not a packaging convenience.

Roles referenced below:

- **`release.yml`** (P4, dist-owned) — tag-driven `cargo-dist` matrix → SHA-256 checksums + SLSA
  provenance + installer artifacts (`curl|sh`, PowerShell, npm, Homebrew) + self-update receipt →
  GitHub Release, and pushes the **Homebrew tap**. It is regenerated by `dist generate`, so it is
  **never hand-edited** (`dist plan` aborts on divergence). It does **not** publish npm.
- **`publish.yml`** (P5, ours, gated) — `npm publish --provenance` of the dist-built npm package,
  behind the protected `publish` environment (npm is irreversible).
- **`sign-release.yml`** (P4/P7, ours, gated) — cosign keyless `sign-blob` of every release artifact
  + a CycloneDX source SBOM.
- **`docker.yml`** (P6, ours, gated) — separate `buildx` job → multi-arch image on Docker Hub,
  cosign-signed + SBOM + provenance.
- **`ci.yml`** — per-PR gate (must be green to merge).

`cargo install c42` from crates.io is **not** a channel — `c42` has git dependencies (forbidden on
crates.io); the "cargo" channel is **`cargo binstall c42`** (pulls the signed Release binary).

All publishing runs only on a signed semver tag; the npm / image / signing jobs run inside the
protected `publish` GitHub Actions environment (required reviewers; environment-scoped secrets).
Every action in the workflows we own is **pinned by commit SHA** — `release.yml` is dist-owned and
uses tag-pinned actions, so **re-pin it with `pinact run` after every `dist generate`**.

## Build & test (locally)

```sh
cargo fmt --all -- --check
cargo clippy --all-targets -- -D warnings
cargo test
cargo build --release           # target/release/42ctl
docker build -t 42ctl:dev .     # minimal non-root image (no push)
```

CI (`.github/workflows/ci.yml`) runs the same plus `cargo audit` + `cargo deny` + `gitleaks` and a
no-push Docker build. Must be green to merge.

Before tagging, run the dry-run preflight — it validates the release would succeed without
publishing anything:

```sh
sh scripts/release-dryrun.sh v0.2.0
```

## Cut a release (gated — wired in P4–P8)

Releases are **tag-driven** and run through the protected `publish` CI environment with required
reviewers. The release pipeline is one signed tag in, six verified channels out.

### 1. Prepare on `main`

```sh
# Bump the crate version to match the tag you intend to push.
#   - edit `version = "X.Y.Z"` in Cargo.toml  (the crate is `c42`; the bin stays `42ctl`)
cargo build --release            # refresh Cargo.lock for the new version
# Update the changelog and commit on main.
git add Cargo.toml Cargo.lock CHANGELOG.md
git commit -m "release: vX.Y.Z"
git push origin main
```

`42ctl version` prints `42ctl X.Y.Z (<commit>)` — the `X.Y.Z` comes from `CARGO_PKG_VERSION`, so the
Cargo version MUST equal the tag (minus the leading `v`). The dry-run script enforces this.

### 2. Push a signed semver tag

```sh
git tag -s vX.Y.Z -m "42ctl vX.Y.Z"
git push origin vX.Y.Z
```

The signed tag is the single trigger for every gated publish job. No manual `cargo publish` /
`npm publish` / `docker push` is ever run by hand.

### 3. `release.yml` (cargo-dist) — binaries + source channels

On the tag, `cargo-dist` runs the cross-platform **matrix build**, then in order:

1. **Matrix build** — `x86_64`/`aarch64` for linux (gnu + musl), macOS (Intel + Apple Silicon),
   and Windows; plus the generated `curl | sh` and PowerShell installers.
2. **Checksums** — a `SHA256SUMS` (and per-artifact `*.sha256`) file over every binary/tarball.
3. **cosign sign (keyless)** — every release artifact is signed with sigstore keyless signing using
   the workflow's GitHub OIDC identity (no long-lived key):

   ```sh
   # what the job runs, conceptually, per artifact:
   cosign sign-blob --yes \
     --output-signature 42ctl-x86_64-unknown-linux-gnu.tar.gz.sig \
     --output-certificate 42ctl-x86_64-unknown-linux-gnu.tar.gz.pem \
     42ctl-x86_64-unknown-linux-gnu.tar.gz
   ```
4. **SLSA provenance + SBOM** — build provenance is attested for the binaries
   (`actions/attest-build-provenance`, queryable later with `gh attestation verify`), and a
   per-release SBOM (CycloneDX/SPDX, e.g. `cargo cyclonedx` or `anchore/sbom-action`) is attached
   to the GitHub Release.
5. **Publish**, all triggered by the same tag / its published Release:
   - **GitHub Releases** (`release.yml`) — binaries, installers, `SHA256SUMS`, SLSA provenance.
   - **Homebrew tap** (`release.yml`) — `cargo-dist` commits the formula to `Univers42/homebrew-tap`
     (needs `HOMEBREW_TAP_TOKEN`).
   - **npm** `@universe42/42ctl` (`publish.yml`, gated) — `npm publish --provenance` of the dist-built
     npm package, in the protected `publish` environment (`NPM_TOKEN`):

     ```sh
     npm publish --provenance --access public ./*-npm-package.tar.gz
     ```
   - **cosign signatures + SBOM** (`sign-release.yml`, gated) — `cosign sign-blob` every asset →
     `*.sig`/`*.pem`, plus a CycloneDX source SBOM, uploaded back onto the Release.
   - **Docker Hub** (`docker.yml`, gated) — multi-arch image, cosign-signed + SBOM + provenance.
   - **cargo** — no crates.io publish (git deps); users get it via `cargo binstall c42` from the
     Release.

### 4. `docker.yml` (buildx) — the image, separately

A **separate** `buildx` job (`docker.yml`) builds `deploy/Dockerfile.dist` for
`linux/amd64,linux/arm64`, pushes to Docker Hub, then cosign-signs the digest and attaches an SBOM
+ provenance:

```sh
# what the job runs, conceptually:
docker buildx build --platform linux/amd64,linux/arm64 \
  -f deploy/Dockerfile.dist \
  -t docker.io/$DOCKER_LOGIN/42ctl:vX.Y.Z -t docker.io/$DOCKER_LOGIN/42ctl:latest \
  --provenance=true --sbom=true --push .
cosign sign --yes "docker.io/$DOCKER_LOGIN/42ctl@${DIGEST}"
```

Image runtime is `FROM scratch` + static musl binary, non-root (`USER 65532`), CA bundle only.

### 5. Verify each channel on a clean machine (Definition of Done)

Run the user-facing verification from `SECURITY.md` against each channel — see
`docs/RELEASE-DOD.md` for the full checklist. At minimum:

- Installed `42ctl version` matches the tag and the release commit, on a clean machine, per channel.
- `cosign verify` / `cosign verify-blob` passes for the artifacts and the image.
- `gh attestation verify` (or `slsa-verifier`) confirms SLSA provenance.
- `sha256sum -c SHA256SUMS` passes.
- `npm audit signatures` confirms npm provenance.
- A **deliberately corrupted** artifact is **refused** by the installer / verifier.

## Credentials

`NPM_TOKEN`, `DOCKER_LOGIN`, `DOCKER_PAT`, and `HOMEBREW_TAP_TOKEN` live **only** as
environment-scoped GitHub Actions secrets on the protected `publish` environment — never printed,
committed, or baked into an image. Prefer credential-free **OIDC** where the registry supports it
(npm trusted publishing; cosign keyless already uses OIDC and needs no key at all). There is **no**
`CARGO_REGISTRY_TOKEN` — `c42` is not published to crates.io (git deps); the cargo channel is
`cargo binstall`.

| Secret | Used by | Scope | Preferred replacement |
|---|---|---|---|
| `NPM_TOKEN` | `publish.yml` (npm publish --provenance) | publish env | npm OIDC trusted publishing |
| `DOCKER_LOGIN` | `docker.yml` (login + image namespace) | publish env | — (username) |
| `DOCKER_PAT` | `docker.yml` (registry auth) | publish env | short-lived PAT, rotate on schedule |
| `HOMEBREW_TAP_TOKEN` | `release.yml` (push formula to the tap repo) | publish env | fine-grained PAT, `contents:write` on the tap |

### Rotate a publish credential

The rotation shape is always **revoke the old → mint the new → update the environment secret**, then
verify a dry-run/release picks it up. Never delete the old before the new is in place if a release is
mid-flight; otherwise revoke-first.

Set a secret (CLI):

```sh
gh secret set NPM_TOKEN   --env publish --repo Univers42/42ctl < /dev/stdin   # paste, no echo
gh secret set DOCKER_PAT  --env publish --repo Univers42/42ctl
gh secret set DOCKER_LOGIN --env publish --repo Univers42/42ctl
```

- **npm (`NPM_TOKEN`)** — *prefer migrating off it entirely:* configure **OIDC trusted publishing**
  for `@universe42/42ctl` in the npm package settings (link the `Univers42/42ctl` repo +
  `release.yml`), then delete `NPM_TOKEN`. If a token is still required: revoke the old token at
  `npmjs.com → Access Tokens` → create a new **Automation / Granular** token scoped to publish
  `@universe42/*` → `gh secret set NPM_TOKEN --env publish`.
- **Docker Hub (`DOCKER_PAT`)** — revoke the old PAT at `hub.docker.com → Account Settings →
  Security → Access Tokens` → create a new PAT with **Read/Write** scope → `gh secret set
  DOCKER_PAT --env publish`. If the publishing account/namespace changed, also update
  `DOCKER_LOGIN` (it doubles as the image namespace in `docker.yml`).
- **Homebrew tap (`HOMEBREW_TAP_TOKEN`)** — revoke the old PAT in `github.com → Settings →
  Developer settings → Fine-grained tokens`, mint a new one with **`contents: write`** on
  `Univers42/homebrew-tap` only → `gh secret set HOMEBREW_TAP_TOKEN --env publish`.
- **cosign** — keyless (OIDC); there is **no key to rotate**. Trust is the workflow identity in the
  Fulcio certificate, which rotates automatically per run.

After any rotation, re-run `sh scripts/release-dryrun.sh vX.Y.Z` (it reports which secrets the
workflows reference) and confirm the next gated job authenticates.

## Revoke / yank a compromised release

A signed + provenance-attested release means a tampered build is already detectable and refused by
any verifying installer — yanking closes the *install* path and the *trust* signal. Move fast and in
this order:

1. **GitHub Release** — the source of the binaries (and what `cargo binstall` + the `curl|sh`
   installer pull). Mark it as a security release (step 4) and, if needed, delete its assets so the
   bad binary can't be fetched:

   ```sh
   gh release delete-asset vX.Y.Z '*' --repo Univers42/42ctl --yes   # or delete the whole release
   ```
2. **npm** — deprecate the bad version (npm forbids unpublish after 72h; deprecate is the supported
   signal):

   ```sh
   npm deprecate @universe42/42ctl@X.Y.Z "compromised — upgrade to X.Y.(Z+1), see GHSA-xxxx"
   ```
3. **Docker Hub** — delete (or re-point) the bad tag so it cannot be pulled:

   ```sh
   # delete the tag via the Docker Hub UI, or:
   curl -s -X DELETE -H "Authorization: JWT ${HUB_JWT}" \
     "https://hub.docker.com/v2/repositories/${DOCKER_LOGIN}/42ctl/tags/vX.Y.Z/"
   ```
4. **GitHub Release / advisory** — mark the GitHub Release as a security release and open a GitHub
   **Security Advisory** (GHSA) describing the affected versions and the fix:

   ```sh
   gh release edit vX.Y.Z --repo Univers42/42ctl --notes "SECURITY: yanked — see GHSA-xxxx"
   ```
5. **Homebrew tap** — revert the bad formula commit in `Univers42/homebrew-tap` (or bump it to the
   fixed version), then **rotate every credential that may have leaked** — follow "Rotate a publish
   credential" above for `NPM_TOKEN` / `DOCKER_PAT` / `HOMEBREW_TAP_TOKEN`. cosign is keyless, so
   nothing to rotate there.
6. **Ship the fix** — cut `X.Y.(Z+1)` immediately through the normal gated release; the SBOM +
   provenance on the new build are the proof of what changed.

Because every artifact and image is signed + SLSA-attested, downstreams that verify (the installers,
`cosign verify`, `gh attestation verify`) will already refuse the tampered build before it runs.
